SCOPO
Questo file serve solo per farti vedere "come appariranno" i dati dentro il database SQLite.
Il database NON sostituisce i file raw: i JSON e i capture restano il materiale sorgente.

RAW CHE RESTANO SU DISCO
- input/cb_manual_runs.jsonl
- input/battle_result_captures/*.bin
- input/battle_result_captures/*.json
- input/battle_result_captures/*.meta.json
- input/loadout_snapshots/*.json

IDEA BASE
Ogni run salvata nel jsonl verra' spezzata in piu' tabelle:
- 1 riga principale in runs
- 0..5 righe in run_champions
- 0..5 righe in run_damage
- 0..molte righe in run_events
- 1 riga in run_raw_assets

COME SI LEGGE MENTALMENTE
Se apri il database, la vista pratica sara' questa:

1. runs
Qui vedi una riga per run: quando, che team, quanto danno, che origine ha.

2. run_champions
Qui vedi chi componeva quella run.

3. run_damage
Qui vedi chi ha contribuito quanto.

4. run_events
Qui vedi la cronologia tecnica dell'incontro.

5. run_raw_assets
Qui ritrovi il ponte verso i file raw originali.

QUERY CHE POI POTREMO FARE
- ultime run salvate
- danno medio per team_name
- danno medio per champion_name
- miglior e peggior contributor
- run con confidence = heuristic
- run senza battle_result_json_path
- confronto stesso team su piu' giorni

MESSAGGIO IMPORTANTE
Il database verra' popolato LEGGENDO i file raw, non cancellandoli.
Quindi il piano corretto e':
- tenere i JSON come archivio sorgente
- creare SQLite come livello analytics
- poter rigenerare il DB in qualsiasi momento se serve
